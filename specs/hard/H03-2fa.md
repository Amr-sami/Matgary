# H03 — 2FA for owners (TOTP + recovery codes)

> Source: `task.md` §7.1 H3

- **Status:** done (2026-06-03) — manual-key enrollment path; QR rendering deferred (see Scope notes)
- **Effort estimate:** 3-4 hrs (actual: ~2 hrs including ditching `otplib` for an in-tree RFC 6238 implementation)
- **Depends on:** H02 (so the migration + login changes land with CI gating)

## Why

Owner credential compromise = full tenant takeover. Single weakest link in the auth model today. Pen-tester (E2) will flag absence. Pair with H09 (session revocation) so a compromised user can recover.

## Acceptance criteria

- [x] Migration adds `users.totp_secret text`, `users.totp_enabled_at timestamptz`, `users.recovery_codes_hash text[]`. (`0026_user_totp.sql`, journal idx 26.)
- [x] `/account/security` enrollment flow — manual base32 secret + `otpauth://` tap-to-add link, verify 6-digit code, on commit display 8 recovery codes once. (**QR rendering deferred** — see Scope notes.)
- [x] Login `authorize`: after password success, throws `TotpRequiredError` if `totp_enabled_at` is set and `totp` is missing; verifies the supplied code (or recovery code) when present; throws `InvalidTotpError` on miss. Both errors carry a `code` field the client parses out of the `json:true` callback `url`.
- [x] Recovery code consumed on use — matched hash spliced out inside the authorize path.
- [x] Regen flow at `POST /api/account/2fa/regenerate` — current password + valid TOTP required.
- [x] Disable at `POST /api/account/2fa/disable` — same auth.
- [x] Rate-limit `auth.totp` 5 / 15 min / user-id; recovery code mismatches consume the same bucket.
- [x] Activity log: `auth.2fa_enable`, `auth.2fa_disable`, `auth.recovery_codes_regenerated` (category `auth`, not the proposed `account` — see Scope notes).
- [x] Owner role only for v1 — gated on `r.ctx.role === "owner"` in `start` + `enable`.
- [x] Unit tests: 16 in `tests/repo/totp.test.ts` covering RFC 6238 vectors, ±1 window tolerance, malformed input, recovery codes.

## Implementation plan

1. Add `otplib` (smaller surface than `speakeasy`).
2. Generate `totp_secret` server-side, 20 random bytes, base32.
3. Recovery codes: 8 × `crypto.randomBytes(5).toString("hex")` (10 hex chars each, grouped `XXXXX-XXXXX`), bcrypt-hashed individually.
4. Login flow split: `authorize` checks password, sets `mfaPending: true` in returned object if 2FA enrolled. JWT callback short-circuits and refuses to issue a session-id JWT until `/api/auth/2fa/verify` succeeds and clears the pending flag.
5. UI: new `/login/2fa` page with code input + "use a recovery code" toggle.

## Out of scope

- WebAuthn / passkeys (later).
- WhatsApp OTP as a 2nd factor (Soft, gated on E4 SMS fallback decision).
- Per-staff 2FA enforcement policy (owner-side toggle is later).

## Risks & gotchas

- JWT trial gate uses a 60 s cache. After 2FA enable, the pending flag must invalidate user-context cache via `bustUserContextCache`.
- TOTP clock drift: accept ±1 window (90 s tolerance total).
- The recovery code "regenerate" step must invalidate old codes atomically — single UPDATE replacing the whole array.
